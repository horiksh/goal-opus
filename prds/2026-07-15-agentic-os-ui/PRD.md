## 0. One-paragraph summary

**agentic-os** today is a headless, file-based, Claude-Code-native CLI that runs a bounded
self-improving loop (maker agent → independent verifier → land-gate → mandatory write-back)
and records everything to local files under `.agentic-os/`. This PRD defines a **local web
"Mission Control"** — a loopback-only dashboard that reads those files live and makes the
system's four mechanics **glanceable, honest, and beautiful**: the **skills installed**, the
**tokens available vs used**, the **running goals** and their lifecycle, and each goal's
**% toward its rubric-defined "done."** "Done" for this product means: *a fresh operator,
watching only the dashboard during a live run, can correctly name the loop's current stage,
read an honest non-monotonic % and watch it move as criteria flip, read token burn against
budget, see what's installed, and notice within seconds when a run stalls or gets flagged —
and it passes vision-verify against a frozen design direction, is accessible, and never
renders stale data as live nor exposes the loopback server.*

## 1. Working-backwards press release

**agentic-os Mission Control ships — the self-improving loop, finally visible.**

For the solo operator who runs agentic-os, watching a run used to mean tailing JSON and
running `status` in a terminal. Mission Control changes that. Run `agentic-os dash`, and a
browser opens on `127.0.0.1` to a living view of your agent home. The **loop itself** is
drawn as a small flow — maker → verifier → land-gate → write-back — with a "you are here"
marker that moves as the current iteration progresses. A **ring** shows each goal's progress
as *criteria passing / total*; it fills as the independent verifier confirms criteria, and —
honestly — it can drop if a criterion regresses. A calm, always-on **token meter** burns down
against your budget and tells you, per iteration, exactly where the tokens went. Your
**installed skills** are cards you can actually read: what each is, its version and source,
whether it's enabled, what it's allowed to touch, and what it costs you in standing context.
When a run stalls, gets flagged for a gamed pass, or nears its budget, that one thing rises to
the foreground; everything else stays quiet. It's local, it's fast, it's yours, and it never
pretends a frozen screen is a live one.

**FAQ (a skeptic's questions):**

1. *Isn't this just a status page with charts?* No. The differentiator is **honesty under a
   real agent loop**: torn-read-safe reads of a file the CLI rewrites mid-iteration, a
   progress number that can go **down**, crash detection, and staleness that's shown, not
   hidden. Every analog we studied fails at least one of these (Airflow's lying spinner,
   GitHub's silent log stall, OpenAI's hours-late meter, AutoGPT's self-graded "done").
2. *"Visionary" and "calm dashboard best-practice" contradict each other — which wins?* Both.
   "Visionary" here is a **novel, live, information-rich rendering of the loop mechanic
   itself** with **functional-only motion** (every animation maps to a real state change) and
   **one focal animation at a time**; "calm" is delivered by **exception-surfacing** — the
   one thing needing attention rises, the rest holds still. [VERIFIED+SRC https://www.nngroup.com/articles/animation-purpose-ux/]
3. *Can it control the loop, or only watch?* v1 is a **read-only observatory**. Control
   (pause/steer/enqueue, then undo/approve) is **phased in behind write-safety**, because the
   CLI has no lock today and the UI must not fight the CLI for the same files.
4. *Is a localhost server safe?* It binds **loopback-only** and validates the **Host header**
   (loopback binding alone does not stop DNS-rebinding — a real 2024 CVE class), and it
   **re-redacts** secrets at display because the CLI's scrub is explicitly gameable and does
   not cover `STATE.md`/`run-status.json`.
5. *Multiple projects?* v1 is **single-target, many-goals** (matches the engine). A
   multi-target fleet view is a deliberate future-PRD item, not v1.

## 2. Users & jobs-to-be-done

- **Primary user:** the **solo operator** who installed agentic-os into a repo and runs the
  loop (the same person for whom the whole framework was built — Windows-first, local-first).
- **Jobs:**
  1. *"Is my run healthy right now?"* — glance and know the stage, the progress, the burn,
     and whether anything needs me. (Today: `status`, tailing JSON.)
  2. *"Where did my tokens go?"* — see budget vs spend and per-iteration attribution.
     (Today: opaque; the engine's own `/usage` analog is behind a command.)
  3. *"What's actually installed and active in this repo?"* — read the skills/agents/tools
     inventory. (Today: read the `.claude/` directory by hand.)
  4. *"What is this thing even showing me?"* — a newcomer (or the operator months later)
     forms a mental model of the maker/verifier/gate/write-back mechanic. (Today: read docs.)
- **What they use instead today:** the `status` / `status --follow` CLI, `tail`-ing
  `run-status.json` and `run-log.jsonl`, and reading `STATE.md` and `git log` directly.

## 3. Scope & non-goals

**In scope (v1):**
- A **local web app** (loopback server + browser dashboard) that **reads** `.agentic-os/`
  artifacts, `STATE.md`, the `.claude/` skills payload, and git checkpoints, live.
- The **four mandated mechanics** as first-class surfaces: skills installed · tokens
  available/used · running goals + lifecycle · % to goal (rubric-derived).
- The **living-loop** visualization, onboarding/empty-states, accessibility, and the
  security + data-integrity floor.

**Non-goals (explicit):**
- **N1 — No multi-target/fleet view in v1.** Single installed target, many goals. Multi-repo
  aggregation is a future PRD (the engine is single-target). [matches engine architecture]
- **N2 — No full control surface in v1.** v1 observatory is **read-only**; control is a
  later phase behind write-safety. (Rationale: no CLI lock exists to coordinate on — verified.)
- **N3 — No cloud, no multi-user, no auth server, no telemetry off the machine.** Local only.
- **N4 — Not a replacement for the CLI.** It's a lens; the CLI stays the engine.
- **N5 — Does not re-implement or move the loop's logic into the UI.** The UI never grades,
  never runs the maker/verifier, never writes `criteria.json`. (Maker≠grader invariant.)
- **N6 — No new analytics database / ETL tier.** Read the files directly. (Langfuse's
  self-hosted 12–24h trace lag traces to exactly this tier. [VERIFIED+SRC https://github.com/orgs/langfuse/discussions/9243])

## 5. Success criteria & evals (Hamel levels)

**L1 — assertions, every change (machine-checkable):**
- Server **binds 127.0.0.1 only**; a request with a non-localhost **Host header is rejected**.
- **Torn-read fixture:** a truncated/partial `run-status.json` → reader returns last-known-good,
  no crash, no "no run."
- **Crash-marker fixture:** `in_flight` present + no `session_end` → UI state = "crashed"
  (distinct from running and from empty).
- **Non-monotonic fixture:** a run-log where criteria_passing **decreases** → the progress ring
  **drops** (no high-water lock).
- **Zero-state fixture:** Default-FAIL 0/M → honest "not started", no animated fill.
- **Color-independence lint:** every status token renders an **icon + text label** (no
  color-only state) — automated DOM assertion.
- **Virtualization assert:** a 10k-line `run-log.jsonl` → bounded DOM node count; no full-file
  DOM dump; interaction stays responsive.
- **Reduced-motion assert:** with `prefers-reduced-motion: reduce`, animations are replaced by
  static fallbacks (media query honored).
- **Secret-redaction fixture:** a fake secret in `STATE.md` → **not** present in rendered DOM or
  any export.
- **Loopback-write guard:** UI process opens the target's files **read-only** in v1 (no write
  handle to CLI-owned files).

**L2 — human + model judge on traces (+ vision-verify):**
- **Vision-verify:** re-captured screenshots of each slice, judged by Reading them against the
  **frozen design direction + BV-list** (verifier names the image files it viewed; a BV
  violation fails the run).
- **Glanceability judge:** a fresh operator, shown the live dashboard for ~2 seconds, can name
  the current loop stage / whether progress is moving / whether anything needs attention.
- **Mental-model judge:** a newcomer, given only the dashboard + glossary, can correctly
  explain what maker/verifier/land-gate/criteria-% mean.
- **Reduced-motion aesthetics judge:** the reduced-motion variant looks *intentional*, not
  broken.

**L3 — live A/B (mature stage):**
- **Adoption:** the operator uses the dashboard (not the CLI) as the primary "how's my run?"
  surface across real runs.
- **Time-to-notice:** median time to notice a stalled/runaway/flagged run drops vs the CLI
  baseline.

## 8. Final acceptance test (one "ship it" gate)

**A fresh operator, given only the running dashboard (no terminal), during a live ≥2-goal
run on `D:\horil\agentic-os`, can:**
1. Name the loop's **current stage** (maker/verifier/land-gate/write-back) and which goal is
   active — from the living-loop view.
2. Read **% to goal** and watch it **move as criteria flip** — including watching it **drop**
   when a criterion regresses (non-monotonic honesty).
3. Read **tokens used vs budget**, the **per-iteration burn**, and the **at-current-rate**
   forecast — always-on, never behind a command.
4. See the **installed skills** inventory with version/scope/enabled/permissions.
5. **Notice within seconds** when a run **stalls, is flagged (gamed pass), or nears budget** —
   that one thing rises to the foreground.

operator), not a mock-graded unit — the same shape as the engine's own §8 live acceptance.

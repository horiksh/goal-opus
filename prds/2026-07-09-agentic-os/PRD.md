## 0. One-paragraph summary

The **agentic OS** is a portable, Claude-Code-native framework that turns any git repository into a
**self-improving agent home**: it installs the goal-opus loop (maker → independent verifier → gate →
mandatory write-back), a plain-file memory layer (STATE.md + `memory/`), and — the new piece — an
**orchestrator** that sequences multiple goal-opus runs over that shared memory, bounded by hard
iteration/token budgets, controllable mid-run (kill-switch, steer), resumable after a crash, and
reversible when an autonomous change is wrong. It is built and dogfooded by a **single solo
operator** on **Claude Code for Windows**; portability across repos and OSes is a design constraint,
not a v1 gate. **"Done" for v1** = a *runnable* orchestration layer: from a fresh clone, the operator
bootstraps the framework into a TARGET repo, enqueues goals, and watches the orchestrator run them to
their Default-FAIL rubrics over shared file memory — safely, boundedly, resumably.

## 1. Working-backwards press release

**It already works.** A solo developer opens a terminal in any repo, types `agentic-os init`, and the
repo becomes a self-improving agent home: a `.claude/` with the goal-opus loop, a seeded STATE.md,
and an orchestrator. They enqueue two goals (`agentic-os goal "…"`), type `agentic-os run`, and walk
away. The orchestrator runs each goal through goal-opus — maker builds, an independent verifier
grades against a Default-FAIL rubric, the gate flips criteria only on evidence, and every run writes
its lesson back to memory before the next begins. A hard `max_iterations` and a token budget cap the
whole session; a `STEER.md` redirects it mid-run; an `AGENT_STOP` file halts it instantly. The laptop
sleeps; on wake, `agentic-os resume` continues from the last git checkpoint without redoing finished
work. When a run declares a bad change "passing," the operator sees it in the run-status file, and
`agentic-os undo` reverts it. Nothing about the operator's setup was special: it ran on Windows,
against their own repo, with no cloud dependency.

**FAQ (a skeptic's questions):**
- *Isn't this just goal-opus?* No — goal-opus runs **one** goal. The OS is the layer **above**:
  it sequences many goals over shared memory, owns the run-lifecycle state machine, budgets, control
  sentinels, resumability, and rollback. (This composition is design — see §4d.)
- *Why not a cloud agent platform?* v1 targets the solo operator's actual setup (Windows, Claude
  Code, local). Cloud routines/durable execution are a **documented future phase**, not v1.
- *How is it "self-improving" if a human seeds the rubric?* The human supplies the *what* (a
  Default-FAIL rubric); the loop supplies the *how* and writes confirmed lessons back to skills/memory
  so the next run inherits them. v1 automates write-back **manually-triggered**; automated
  consolidation ("dreaming") is deferred (§3).
- *You said the "done" gate is objective — is it trustworthy?* Partly. The Default-FAIL contract is
  an **anti-drift guardrail, not a security boundary** (its own source says so). That is exactly why
  v1 ships an **undo/rollback** path and an optional approve-before-land gate (§4b U33).
- *Portable to macOS/Linux?* Designed for it (plain files, git), but the **install mechanic is
  verified only on Windows in v1**; other OSes are a stated seam, proven by a real install test before
  any portability claim is made (§3, §4b U24/U25).

## 2. Users & jobs-to-be-done

**Primary user: the solo operator** (this repo's owner) running and steering their own agent home.
Jobs: *(a)* stand up a self-improving loop in any repo in one command; *(b)* hand off multi-goal,
multi-hour work and walk away, staying able to watch/steer/stop from a glance; *(c)* trust that a bad
autonomous change is bounded, visible, and reversible; *(d)* have every run leave the next run
smarter. **What they use today:** goal-opus one goal at a time, driven by hand; `/loop` for intervals;
STATE.md edited manually. The OS makes the *orchestration* between goals structural.
**Secondary (design-constraint only, not a v1 user): adopters** who might clone the framework — served
by portability seams, not by v1 onboarding docs (§3, out of scope).

## 3. Scope & non-goals

**In scope (v1 — the runnable orchestration layer):**
- The **orchestrator control loop** over a **single active TARGET**, sequencing goal-opus runs over
  shared file memory (§4d). **[D]**
- **Single-threaded goal execution** — one goal-opus run at a time (matches solo operator; honors the
  verified "fan-out is expensive / single-thread by default" stance). **[V/R]**
- Hard **`max_iterations` + token/cost budget** as orchestrator-level stop conditions, with cumulative
  spend surfaced. **[V/R]**
- **Run-lifecycle state machine**: idle · running · paused(rate-limited/offline) · no-progress ·
  success · aborted · crashed→resumable (§4d). **[D]**
- **Control sentinels**: `AGENT_STOP` kill-switch, `STEER.md` mid-run redirect. **[V]**
- **Guaranteed write-back** on success/abort/crash (compounding contract, orchestrator-level). **[V]**
- **Resumability + idempotent replay** after crash/sleep (resume from git checkpoint + run log; never
  double-apply write-back or criteria flips). **[V/R]**
- **Rollback/undo** of a landed autonomous change + optional approve-before-land gate. **[D]**
- **Plain-file memory** with export (git/copy), a secret-scrub path, and a **size-threshold warning**
  (manual consolidation; automated "dreaming" deferred). **[V/A]**
- **Windows-first install** (`init`/`install`/`uninstall`/`upgrade`) with a **copy fallback** where
  NTFS junctions are unavailable; schema-version field on STATE.md/criteria. **[V/A]**
- **Local `/loop` interval scheduling** as the v1 trigger surface. **[R]**
- **Pull observability** (tailable run-status file + sentinels) **+ minimal push notification** on
  finish/block/budget/decline. **[R/A]**
- **OS self-test** (smoke test of the run loop + a migration + an install round-trip). **[D]**
- **Least-privilege tool allowlists** (verifier read-only) + **classifier-decline branch** (route to
  fallback/human, never retry-as-error). **[V/A]**

**Non-goals (v1 — deferred, each with the reason; NOT silently omitted):**
- **Concurrent multi-goal fan-out** and its concurrency-safety machinery (file-ownership
  partitioning, lock-claiming, stuck-task reconciliation — R3 U16–U19). *Reason:* solo operator +
  single-thread-default; shipping the machinery and the default simultaneously is incoherent (R4 §5.3).
  The **banned outcome B9 still applies** the moment concurrency is ever enabled.
- **Cloud / event-triggered autonomy** (cron routines, webhooks, GitHub-event triggers, durable
  execution surviving laptop sleep). *Reason:* no verified research (R1 gap #1), and it contradicts the
  local Windows v1 (R4 §5.4). **v1 runs only while the machine is awake** — stated plainly.
- **Automated memory consolidation / "dreaming."** *Reason:* no surviving research substantiates an
  automated loop (R1 gap #4); v1 = manual write-back + a size-threshold warning.
- **Agent-Teams-based multi-agent coordination.** *Reason:* the only vendor-native substrate is
  EXPERIMENTAL and its own docs say "do not depend on its stability" (R4 §5.2). Deferred until GA.
- **Multi-TARGET registry / switching.** *Reason:* v1 = one active target; a registry is a later phase
  (R4 §1.5).
- **Adopter onboarding docs, OSS license/contribution/versioning.** *Reason:* v1 user is solo (R4 §1.7).
- **Cross-platform (macOS/Linux) install.** *Reason:* v1 verifies the Windows mechanic only; other OSes
  are a documented seam (R4 §5.1).
- **Any status dashboard / GUI.** *Reason:* v1 observability is tail/watch + a status file; a dashboard
  would be a **UI slice requiring `/design-direction`** before its own goal-opus run — out of v1.

### 4d. The orchestrator control loop  — **[D] design, no direct research support (R4 §1.1, highest-severity gap)**

R4 correctly flagged that the literal product — "a runnable orchestration layer" — had its parts
researched (the *single*-goal builder/evaluator loop, memory, gates, states) but never its own
control flow. This section is **design composed from R1's verified primitives**, and it is flagged as
design: it is validated not by a citation but by its Phase-1 build gate (§7). It reuses, does not
reinvent: goal-opus is the verified single-goal loop; `/goal`'s Stop-hook completion check, cwc's
headless "exit on contract/no-change/budget," and the Ralph Stop-hook re-feed are the verified loop
primitives it composes.

```
The orchestrator = a bounded, resumable loop over a goal QUEUE on ONE active TARGET:

  BOOTSTRAP (once): U1 init · U2 git-precondition · U27 dual-shell · U8 schema-version
  ── enter run loop ──────────────────────────────────────────────────────────────────
  each iteration:
    1. POLL sentinels           → AGENT_STOP present? halt (U11, write-back U32) → EXIT
                                → STEER.md present?  fold in once, then delete (U11)
    2. CHECK budgets            → max_iterations or token budget hit? → state=aborted, write-back → EXIT (U12/U13/B3)
    3. PICK next goal           → queue empty? → state=idle, clean no-op EXIT (U3)
    4. RUN one goal-opus goal   → single-threaded (U14); its own Default-FAIL rubric (B2/B5)
    5. GATE the result          → success | needs-work | no-progress(U15) | decline(U22→fallback/human, B10)
    6. WRITE BACK (U32/B7)      → STATE.md/memory (idempotent U31), git checkpoint, run log (U10), status file (U9)
    7. LAND-GATE (U33)          → optional approve-before-land; else land with an undo pointer recorded
    8. NOTIFY (U35) on terminal events; re-enter
  ── resumability (U30) ──: on start, if a partial run is detected, rehydrate from git checkpoint +
     run log + locked contract and re-enter at the last incomplete step — never restart from zero;
     all steps 6–7 are idempotent so replay is safe (U31).
```

Run-lifecycle states (the machine steps 1–8 transition between):
`idle → running → {paused(U4) | no-progress(U15) | success | aborted | crashed}`, with
`crashed/paused → running` via resume (U30). **This state machine and its per-iteration poll order are
the v1 deliverable's spine; Phase 1's gate (§7) is what proves the design.**

## 5. Success criteria & evals (seed `criteria.seed.json`)
- **L1 (assertions, every change):** `tools/rubric_check.py` passes on any generated `criteria.json`;
  the orchestrator **refuses to start a goal that has no Default-FAIL rubric or no `max_iterations`**
  (B3); a write-back artifact exists after **every** run outcome including a simulated crash (B7); a
  secrets scan over memory files finds none (B4); `agentic-os` core verbs exist and exit-code
  correctly (U34).
- **L2 (human/model judge on traces):** a **fresh-context verifier** (goal-verifier posture), given
  only artifacts (status file, run log, git history), confirms the orchestrator sequenced ≥2 goals
  over shared memory with correct state transitions and a write-back per run; a **deliberately gamed
  "false pass"** is caught by the land-gate and reverted by `undo` (U33/B5/B11); the run-status file is
  human-legible.
- **L3 (live, maturity):** the operator runs a real ≥2-goal session on a real TARGET end-to-end on
  Windows; a **crash injected mid-run resumes** from checkpoint with **no double-applied write-back**
  (U30/U31); an install round-trip (`init`→`uninstall`) leaves no residue (U26/U36).

## 8. Final acceptance test (one objective "ship it" gate)
> **From a fresh clone on Windows**, the operator runs **one bootstrap command** to install the
> framework into a TARGET repo, enqueues **two goals**, and the orchestrator runs **both goal-opus
> loops to their Default-FAIL rubrics over shared file memory**, bounded by `max_iterations` **and** a
> token budget, **writing back after each**. Mid-run, an `AGENT_STOP` **halts within one iteration**;
> `agentic-os resume` **continues from the last git checkpoint without redoing completed work**; and a
> change that landed "passing" but is wrong is **reverted by `agentic-os undo`**. Verified two ways:
> (1) re-running purely from the **status file + git log** reproduces the run's state, and (2) a
> **fresh-context verifier**, given only artifacts (no orchestrator reasoning), confirms the state
> transitions, the per-run write-backs, and the undo — from evidence alone.

## 9. Risks & open questions
- **[ASSUMPTION] Windows install mechanic (U24/U25).** NTFS junctions need privilege and can fail;
  the user/project config-merge precedence is R1's unresolved gap #2. *Verify:* a **real install test**
  in P0 (junction path + copy-fallback path); confirm precedence against Claude Code docs. If it
  doesn't hold, narrow the portability claim.
- **[ASSUMPTION] Scheduling & laptop-sleep (U28/U29).** v1 decision = local `/loop`, runs-while-awake;
  cloud durability deferred. *Risk:* if unattended overnight autonomy is actually required, this forces
  the deferred cloud phase forward. Stated, not hidden.
- **[REPORTED-only, load-bearing] Concurrency/idempotency/budget/scheduling (U8,U12,U16–U19,U28,U31).**
  Their only support is unverified `[R]`. v1 either defers them (concurrency) or **seeds them as
  build-time rubric criteria to prove**, never asserts them as fact. *Verify:* replay test (U31),
  budget-stop test (U12/U13).
- **[UNCOVERED analog — accept-risk, named] MAS semantic failures** (inter-agent misalignment,
  information withholding, weak verification; R2 survey, R4 §3) are **not mined**; v1 mitigates by
  single-threaded-default, but the risk is recorded, not closed. **OpenHands #8630** is an **un-mined
  pointer** (R4 §3). Both are explicit debt, not silent gaps.
- **[CAVEAT] Single-vendor evidence & self-reported benchmarks** (header). The whole harness spine is
  Anthropic-corroborates-Anthropic. Independent validation is an open external-risk item.
- **[CONTRADICTION resolved, tracked] Agent Teams dependency (R4 §5.2).** U16–U20/U23 originally leaned

How the agent works — the self-improving goal loop

One fixed model (Opus 4.8), three separated roles. The model never gets smarter — the files around it do.
Orchestrator

1 · Define done FIRST

criteria.json — every criterion starts failing, each with an executable check, plus banned outcomes (instant fail) and a hard iteration cap. You sign this off — it is locked from then on.

→
Maker · builds

2 · Make

Walks the 7-rung decision ladder (YAGNI → reuse → stdlib → … → minimum viable) before writing code, then must enumerate every criterion: satisfied / not-satisfied. Never grades itself, never touches the rubric.

→
Verifier · grades

3 · Adversarial verify

Fresh context, no Write/Edit tools, never sees the maker's notes. Executes every check, tries to refute, fails both over-engineering and under-implementation. UI work is judged by looking at re-captured screenshots.

→
Orchestrator

4 · Gate

Only the orchestrator flips a criterion to passing — and only on a verifier pass with evidence. Any fail → structured diff back to the maker. All pass → done. Cap hit → honest abort, never a fake success.

5 · Write-back — mandatory, every run

Success, abort, or cancel alike: lessons are distilled into general rules → STATE.md; confirmed failures append to the skill's own Known failure modes. Then everything commits to git.

Why this compounds

The next run reads what the last run learned — rules ride along in every maker packet, failure modes are consulted before each rubric. Wrong lessons get corrected as dated entries, and every self-edit is a reviewable git diff. Six runs so far; the loop has already caught and fixed one of its own false beliefs.

Maker ≠ grader. The builder never judges its own work.
Default-FAIL. Nothing passes without executed evidence.
Write before walking. A run without write-back isn't finished.
Verified by looking. Visual work is graded from screenshots, never prose.
Home ≠ target. Product code lives in each goal's target repo, never here.

Operating manual

Rules of thumb — using it efficiently

The habits that keep the system compounding instead of drifting.

One home, many targets

  • Never clone this repo per project — clones fork the self-improving skill and splinter its learning.
  • Every build goal declares target: <path>; product code + its own STATE.md live there.
  • The home keeps only the system, memory, and run evidence.

Where memory lives

  • STATE.md — verified facts, decisions D1–D8, open failures, resume pointer.
  • SKILL.md sections — known failure modes, anti-patterns, eval suite, run log (append-only, dated).
  • git history — every self-edit is a diff you can audit and revert.

Your control points

  • Rubric sign-off (Phase 1) — the main steering wheel.
  • Design direction — taste enters here, once, then freezes.
  • Slice demos — motion & feel feedback.
  • Everything else runs hands-off.

Research depth

  • /prd judges for itself when to escalate to deep research — and says which depth it chose.
  • /deep-prd forces it, and fails loudly if the plugin is missing — never a silent downgrade.

Settled — don't re-litigate

  • No RAG / vector index; no maker-side code-graph — grep over live files won on evidence (0.92 vs 0.83); details in docs/context/11.
  • No enforcement hooks — they can veto work the rubric requires.
  • Minimalism ladder + completeness guard are live in every run.

Gotchas

  • New skills/agents register at session start — restart after installing, retry once if an agent isn't found.
  • Goals into a fresh session, not the one that scaffolded the files.
  • First UI slice E2E-tests vision-verify — check the verdict names the image files it viewed.
